What We Learned

What did not work well enough
  • The model could sometimes do one simple step, but not a whole chain reliably
  • Training it more was not the miracle fix
  • Some smaller test numbers looked better than the real outcome
  • So “just trust the model” was not enough
What started helping
  • Make a short list of possible next moves
  • Use a checker to test whether each move is valid
  • Try the better-looking moves first
  • Stop once a time or effort limit is reached
So the project changed direction. Instead of hidden “thinking”, it became visible step-by-step problem solving. The central thesis is simple: pure-neural multi-step execution was weak on this setup, while verifier-guided search with ranking was strong and inspectable under bounded compute.
This is the turning point of the project narrative. We should not oversell single-step. Single-step was useful as scaffolding, but not as the final proof. The thing that actually became strong was explicit search with verification and ranking.

The critical message is that search control mattered much more than pure next-step generation quality. No-ranker spends budget badly. Heuristic and learned ranking both massively improve efficiency. How It Works

make options → sort them → check them → keep trying → show the result

Inside

  • make a short list of possible next steps
  • rank the list
  • check whether each step is valid
  • keep searching until budget runs out

Formal names: candidate generation, branch ranking, verification granularity, and compute-bounded search.

Outside

  • see the checked option shown here
  • change the effort limit
  • see the path taken
  • see why it failed
One active research question is optimal verification granularity: how often you should check intermediate steps under a fixed compute budget.
The nice thing here is that the research stack and the product stack line up. The same components measured in the experiments are the components exposed in MCP. That makes the deck feel grounded instead of speculative.

Academic Ideas Inside It

Test-time scaling

Instead of only training more, spend smart compute while solving.

Verifier-guided search

Do not trust one guess. Generate options and let a checker reject bad ones.

Verification granularity

Decide how often to check the work so you do not waste too much compute.

A simple way to say it:
do a little search, check the work, spend compute carefully, and keep the process visible.

What Is Still Weak

Hard truth
  • Some of the current quality still depends on shortcuts
  • The biggest shortcut is extra structure around where an option came from
  • When we remove that shortcut, harder cases get worse
Latest warning sign

The easy test improved.

The harder test still got worse.

So a nice small result does not automatically mean the real problem is solved.

That hidden gap is exactly what this project helps us see instead of cover up. This also matches a warning from recent work on scaling flaws of verifier-guided search: a system can look strong on easier slices while still depending on shortcuts.
This slide protects the credibility of the whole presentation. We are not hiding the negative results. The learned system is useful, but not yet free from shortcut structure.

What We Need To Prove Next

Keep going if
  • the learned ordering keeps helping on harder tasks
  • most failures still look like effort or search-order problems
  • the system keeps being useful as a product, not just a lab result
Stop if
  • performance collapses when shortcuts are removed
  • the learned ordering stops adding anything beyond hand-made rules
  • the product becomes only pretty charts around hand-made rules
The next step is not another random LoRA run. The next step is a harder transfer test while staying honest about generalization, search-time compute, and shortcut dependence.
Finish with discipline. The system is promising enough to continue, but only if it keeps working under reduced leakage and harder transfer. That makes the roadmap clear and keeps us from drifting back into vague “maybe the weights will learn it” territory.
